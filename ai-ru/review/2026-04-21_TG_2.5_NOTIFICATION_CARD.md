**Статус:** 📋 Design · **Заведён:** 2026-04-23 · **Эпик:** [2026-04-21_TELEGRAM_EPIC.md](2026-04-21_TELEGRAM_EPIC.md)

## 1. Цель

Финальная фаза эпика. Два блока:

- **Карточка уведомления** — заменить plain-text уведомление в Telegram на HTML-карточку с эмодзи по `reasonId`, именем метки, временем и двумя кнопками «✓ Отметить прочитанным» / «🚘 К метке».
- **Phase-5-ready scene registry** — ревью архитектуры и минимальная подчистка, чтобы новая сцена (например, `ParkingRentalScene`) добавлялась просто `@Component` без правок ядра. Разведка 2.5 показала, что ядро уже чистое — останется только вынести hardcoded `"home"` в константу и добавить Javadoc-пример.

## 2. Новый формат уведомления

### Текущий:

```
Audi Q5:
Произошло ДТП
```
(plain text, одна кнопка «отметить прочитанным»)

### После 2.5:

```
🚨 <b>Audi Q5</b> · №145
Произошло ДТП

«Сработала сигнализация, пошла 5 минут назад»

<i>23 марта, 10:35</i>

[✓ Отметить прочитанным]  [🚘 К метке]
```

Структура:
- 1-я строка: эмодзи-по-reason + HTML-жирное имя метки + `· №<seqNumber>`
- 2-я строка: `reasonDescription` (без эмодзи — он уже в заголовке)
- Пустая строка, затем `«<userText>»` в кавычках-ёлочках (если `text` не совпадает с `reasonDescription`)
- `<i>Дата, HH:mm</i>`
- Inline-клавиатура — два ряда или один (зависит от длины):
  - `[✓ Отметить прочитанным]` → `notif:read:<notifId>` (уже есть)
  - `[🚘 К метке]` → `qr_details:open:<qrId>` (уже есть сцена)

### Поведение «отметить прочитанным»

- Как и сейчас: `NotificationFacade.readBy(...)` + `EditMessageReplyMarkup` убирает клавиатуру. Карточка остаётся с пометкой «✓ прочитано» в нижней строке текста.
- Новое: после нажатия «Отметить» текст расширяется — добавляется строка `<i>✓ Прочитано</i>` внизу. Чтобы пользователь видел, что прочитал именно это (особенно если у него много открытых карточек).

### Если `NotificationFacade.shouldShowMarkAsReadButton(id)` == false

(notification уже прочитано, или вне-TG каналом) — кнопку «Отметить» не показываем; кнопка «К метке» остаётся.

## 3. Маппинг reason → эмодзи

Текущий список из `changelog-1.1.xml` (id 1..8):

| id | description | эмодзи |
|----|-------------|--------|
| 1 | Автомобиль мешает | 🚧 |
| 2 | Автомобиль эвакуируют | 🚔 |
| 3 | Произошло ДТП | 🚨 |
| 4 | Открыто окно | 🪟 |
| 5 | Работает сигнализация | 🔔 |
| 6 | Спущено колесо | 🛞 |
| 7 | Мойка завершена | ✨ |
| 8 | Ремонт завершен | 🔧 |

**Реализация:** `tg.reason.emoji.<id>` в `telegram_ru.properties`. Если не найден — fallback `tg.reason.emoji.default` = `🚗`.

Не трогаем БД (схему `reason_dictionary` не меняем) — чтобы не мигрировать. Эмодзи — UI-аспект, живёт в i18n. Когда понадобится админ-редактируемость ([TECH_DEBT A13](../TECH_DEBT.md)) — переделаем.

## 4. Данные для рендера

`NotificationMarkReadScene.renderNotification(TextMessage msg)` сейчас получает `TextMessage` с `text`, `notificationId`, `setting`. Чтобы собрать карточку — нужны `reason`, `qr`, `createdDate`.

**Решение:** Scene по `msg.notificationId` подгружает `Notification` через `NotificationService.findByIdOrThrowNotFound(id)`. Модель `Notification` уже имеет поля `qrId`, `reasonId` — но `reason: ReasonDictionary` и `qr: Qr` могут быть null (зависит от DAO). Проверяем в spec:

- Если `notification.reason` null → показываем emoji `tg.reason.emoji.default` + пустую строку reason.
- Если `notification.qr` null → «Метка» вместо имени.
- Если `notification.createdDate` null → строка с датой не рендерится.

Чтобы гарантированно иметь полные данные, Scene вызовет `notificationService.findWithRelations(id)` — если нет такого метода, придётся добавить. Проверю в имплементации; если проще обойтись существующим `findByIdOrThrowNotFound(id)` — он уже тянет relations — отлично.

- Карточка уведомления рендерится в новом формате; все тесты `NotificationMarkReadSceneTest` зелёные + новые кейсы.
- Эмодзи-ключи в i18n.
- Роутер не содержит литералов scene-key (`HomeScene.KEY` вместо `"home"`).
- `TelegramScene` имеет Javadoc-пример добавления сцены.
- `CarIdApplicationTests` поднимает контекст.
- Docs: `F8_TELEGRAM_BOT.md`, `BF1_TELEGRAM_BOT_UI.md`, мастер-эпик — закрывают 2.5, **и эпик в целом**.
- Мануальный smoke: отправить тестовое уведомление (через `POST /api/report.send` на dev) — увидеть HTML-карточку в TG.

## 8. Что НЕ делаем

- Админский UI для редактирования эмодзи причин ([TECH_DEBT A13](../TECH_DEBT.md)).
- Групп/категорий причин.
- Карточку-превью в списке уведомлений (`NotificationListScene` — просто стала делегировать в `NotificationMarkReadScene.renderNotification`; формат там уже новый).
- Множественный выбор/bulk-действия («отметить всё прочитанным»).
- Изменение форматов других каналов (push, SMS, Zvonok) — они используют `TextMessage.text` plain-string без изменений.
