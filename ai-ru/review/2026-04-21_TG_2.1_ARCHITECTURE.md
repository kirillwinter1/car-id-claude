# Спек 2.1 — Telegram-бот: архитектура + Scene-скелет

**Статус:** 📋 Design · **Заведён:** 2026-04-21 · **Эпик:** [2026-04-21_TELEGRAM_EPIC.md](2026-04-21_TELEGRAM_EPIC.md)

## 1. Цель и не-цели

**Цель.** Ввести архитектуру «сцен», убрать известные архитектурные смрады в Telegram-боте ([TECH_DEBT A1–A4](../TECH_DEBT.md)):

- A1 — циклическая зависимость `TelegramBotService ↔ TelegramLogicService` + `@Setter` на классе;
- A2 — монолитный `onUpdateReceived` (авторизация + callback-роутинг + text-switch в одной функции);
- A3 — `TelegramBotService` напрямую ходит в `NotificationRepository`;
- A4 — магические строки разбросаны по коду.

Дополнительно: **удалить админ-меню** (`/registerUserCount` и пр.) — мониторинг закрыт Prometheus/Grafana на проде.

**Видимое поведение для пользователя не меняется** — за исключением исчезновения админ-меню у админов. Это preserve-behaviour рефакторинг.

**Не-цели (делаем в 2.2+):**

- Новые экраны, HTML/эмодзи/карточки уведомлений — 2.2, 2.5.
- Deep-link онбординг — 2.4.
- Многошаговые формы (scene-state в БД/callback) — 2.3.
- Работа с Phase 5 доменами — за рамками эпика.

## 3. Ключевые интерфейсы

### TelegramTransport

```java
public interface TelegramTransport {
    void send(SendMessage message);
    void edit(EditMessageText message);
    void edit(EditMessageReplyMarkup markup);
    void sendDocument(SendDocument document);
    void sendFeedback(Long channelId, String text);   // для MessageService.asyncSend feedback flow
}
```

Реализуется `TelegramBotService`. Все методы конвертируют `TelegramApiException` в `MessageNotSendException` (сохраняем текущий контракт).

### TelegramScene

```java
public interface TelegramScene {
    String key();                                     // "home", "qr", "notif", ...
    boolean canHandle(TelegramUpdateContext ctx);     // для текстовых команд: совпал ли текст с action сцены
    SceneOutput render(TelegramUpdateContext ctx);    // первичный рендер (например, открыли меню)
    SceneOutput handle(CallbackData data, TelegramUpdateContext ctx);  // нажали inline-кнопку
}
```

### SceneOutput

```java
public record SceneOutput(
    String text,
    InlineKeyboardMarkup inlineKeyboard,      // nullable
    ReplyKeyboard replyKeyboard,              // nullable; для случаев когда нужна reply-клава (main menu)
    boolean editInPlace                       // true — editMessageText; false — новая send
) {}
```

### CallbackData

```java
public record CallbackData(String scene, String action, List<String> args) {
    public static Optional<CallbackData> parse(String raw);   // "qr:pdf:<uuid>" → (scene=qr, action=pdf, args=[uuid])
    public String serialize();                                 // обратно в "scene:action:args"
}
```

Бюджет — 64 байта. Валидация формата, отбрасывание мусора, тесты на граничные случаи.

## 4. Data flow

```
TelegramBotService.onUpdateReceived(Update)
   │
   ▼
TelegramRouter.route(Update)
   │
   ├─ вытащить chatId (callback / message / иначе return)
   │
   ├─ если chatId не привязан в notification_setting:
   │    → TelegramAuthorizationService.handle(update)
   │       • нет телефона в тексте → попросить контакт (sendContact)
   │       • телефон найден → updateTelegramDialogIdByUserId + welcome
   │       • телефон не найден → «Скачайте приложение...»
   │    → return
   │
   ├─ собрать TelegramUpdateContext (userId, user, update, isAuthorized=true)
   │
   ├─ если update.hasCallbackQuery():
   │    data = CallbackData.parse(update.getCallbackQuery().getData())
   │    scene = SceneRegistry.get(data.scene())
   │    output = scene.handle(data, ctx)
   │    renderer.render(output, ctx) → transport.execute
   │
   └─ если update.hasMessage().hasText():
        text = message.getText()
        scene = SceneRegistry.findByTextTrigger(text, ctx)   // canHandle() по тексту
        output = (scene != null) ? scene.render(ctx) : HomeMenuScene.renderUnknown(ctx)
        renderer.render(output, ctx) → transport.execute
```

**Разрыв цикла.** `TelegramRouter` зависит от `TelegramTransport`, `SceneRegistry`, `TelegramAuthorizationService`. Бот (`TelegramBotService`) зависит от `TelegramRouter`. Роутер **не** зависит от бота. Spring-цикла нет, `@PostConstruct`-bootstrap удаляется — остаётся только `TelegramBotService.init()` (регистрация в `TelegramBotsApi`).

**Sender-контракт (внешний API канала).** `TelegramBotService implements Sender`. Внутри:

```java
@Override
public boolean sendNotification(TextMessage message) {
    SceneOutput output = notificationMarkReadScene.renderNotification(message);
    renderer.sendNew(output, message.getSetting().getTelegramDialogId());
    return true;   // ошибка = MessageNotSendException (сохранён текущий контракт)
}
```

`MessageService.asyncSend` и прочий внешний код не меняется.

## 5. Миграция существующих команд в сцены

В 2.1 **все текущие команды работают, тексты те же, поведение то же**. Разница только в том, что вместо switch-case — вызов метода сцены.

| Команда / callback                | Обработчик                              | Ключи из `telegram_ru.properties`                  |
|-----------------------------------|-----------------------------------------|----------------------------------------------------|
| shareContact / phone input (*)    | `TelegramAuthorizationService`          | `tg.auth.request_contact`, `tg.auth.welcome`, `tg.auth.not_registered`, `tg.auth.already_linked` |
| текст «QR-коды»                   | `QrListScene.render` (canHandle=true)   | `tg.qr.list.title`, `tg.qr.list.empty`              |
| callback `qr:list`                | `QrListScene.handle`                    | (edit-in-place пересобирает тот же список)          |
| callback `qr:pdf:<uuid>`          | `QrListScene.handle`                    | (отправка PDF — текст не меняется)                  |
| текст «Временный QR»              | `TemporaryQrScene.render`               | `tg.temp_qr.created`, `tg.temp_qr.error`            |
| callback `notif:read:<uuid>`      | `NotificationMarkReadScene.handle`      | `tg.notification.mark_read` (при renderNotification) |
| неизвестный текст (авторизован)   | `HomeMenuScene.renderUnknown`           | `tg.home.unknown_command`                           |

(*) Авторизация не оформлена как `TelegramScene`: pre-auth flow обрабатывает отдельный сервис, потому что интерфейс `TelegramScene` подразумевает уже авторизованный контекст (`ctx.userId` не-null).

### Ключи `telegram_ru.properties` (финальный список уточняется при реализации)

```
# authorization
tg.auth.request_contact=Введите телефон Вашей учетной записи для авторизации
tg.auth.welcome=<текущий ApplicationConstants.HELLO_TELEGRAM_MESSAGE>
tg.auth.already_linked=<текущий ApplicationConstants.TELEGRAM_ALREADY_HAS_MESSAGE>
tg.auth.not_registered=Скачайте мобильное приложение и зарегистрируйте свой телефон

# home menu (main reply-keyboard)
tg.home.btn.qrs=QR-коды
tg.home.btn.temp_qr=Временный QR
tg.home.unknown_command=Неизвестная команда

# qr list
tg.qr.list.title=QR-коды:
tg.qr.list.empty=<текущий ApplicationConstants.USER_HAS_NO_QR>

# temporary qr
tg.temp_qr.created=<текущий ApplicationConstants.CREATE_TEMPORARY_QR — с плейсхолдерами {0},{1},{2}>
tg.temp_qr.error=внутренняя ошибка при создании временного кода

# notification
tg.notification.mark_read=отметить прочитанным
```

Тексты берутся из текущего кода дословно; плейсхолдеры переводятся в формат `MessageFormat` (`{0}`, `{1}`).

**`ApplicationConstants`** на эти тексты: константы, которые теперь только в properties, удаляем; те, что используются где-то ещё — оставляем (проверится grep'ом при реализации).

## 6. Удаление админ-меню

**Удаляется из кода:**

- `TelegramMenu.admin()` и вся reply-keyboard админа.
- Константы `REGISTER_USER_CMD`, `ACTIVATE_QR_CMD`, `SEND_NOTIFICATION_CMD`, `READ_NOTIFICATION_CMD` (переезжают — никуда; просто исчезают).
- Метод `TelegramLogicService.messageForCommand` и соответствующие ветки switch (вместе с самим `TelegramLogicService`).
- Параметр `isAdmin` в `TelegramBotService.sendMessage(chatId, text, isAdmin)` — становится ненужным (`HomeMenuScene` всегда даёт user-menu).

**Проверяется grep'ом при реализации и удаляется, если больше нигде не используется:**

- `MonitoringService.getRegisterCountToday()`
- `MonitoringService.getActivateQrCountToday()`
- `MonitoringService.getSendNotificationCountToday()`
- `MonitoringService.getReadNotificationCountToday()`

Если методы используются ещё где-то (например, в `/actuator/` endpoint'ах, в какой-нибудь админке веба) — оставляем.

**Не трогаем:**

- Поле `User.isAdmin` — используется в других местах (авторизация админских endpoint'ов).
- Monitoring-метрики (Prometheus counters в `MonitoringService`) — это отдельная ответственность класса.

## 7. Тестовая стратегия

### Юнит-тесты

- **`CallbackDataTest`** — валидные парсы (`qr:list`, `qr:pdf:<uuid>`, `notif:read:<uuid>`), невалидные (пустая строка, без `:`, с лишними `:`, с невалидным UUID в аргументе), `serialize` round-trip.
- **`TelegramMessagesTest`** — все ключи из списка выше присутствуют в properties (fail-fast при реализации новых сцен в 2.2+), плейсхолдеры подставляются.
- **`TelegramRouterTest`** — 
  - Update без message и без callback → no-op.
  - Update с message без привязанного chatId → делегирует в `TelegramAuthorizationService`.
  - Update с callback → `SceneRegistry.get(scene).handle(...)`.
  - Update с известным text → `scene.render(...)`.
  - Update с неизвестным text → `HomeMenuScene.renderUnknown(...)`.
  - Update с неизвестным scene в callback → fallback (лог + отправка «неизвестная команда»).
- **`TelegramAuthorizationServiceTest`** — 
  - Пустой текст → запрос контакта.
  - Телефон не валиден → запрос контакта.
  - Телефон валиден, юзер найден → обновление `telegram_dialog_id` + welcome.
  - Телефон валиден, юзер не найден → «Скачайте приложение...».
  - `existsByTelegramDialogId` уже true → «уже привязан».
- **`HomeMenuSceneTest`** — `canHandle("QR-коды")` → true, `canHandle("Временный QR")` → true, `canHandle("мусор")` → false; `render(...)` возвращает правильную reply-keyboard.
- **`QrListSceneTest`** — пустой список → `tg.qr.list.empty`; непустой → inline-кнопки с UUID; `handle("pdf", uuid)` → вызывает `qrService`/`batchRepository` + `transport.sendDocument`.
- **`TemporaryQrSceneTest`** — успешное создание → `tg.temp_qr.created` с URL; `AppException` → текст ошибки из exception; прочие Exception → `tg.temp_qr.error`.
- **`NotificationMarkReadSceneTest`** — renderNotification → `SendMessage` с `tg.notification.mark_read` кнопкой; handle → `notificationFacade.readBy(...)` + убрать кнопку; если `notification.status = READ` → не показывать кнопку при renderNotification.

### Инфраструктура

- `TelegramTransport` мокается через Mockito везде.
- Real `TelegramLongPollingBot` не стартует в тестах. В `application-test.yml` (или в тестовых `@TestPropertySource`): `telegram.enable=false`.
- `MessageSource` используется реальный (загружает настоящий `telegram_ru.properties`), чтобы тесты ловили отсутствующие ключи.

### Покрытие

Цель — **≥ 80% JaCoCo line coverage пакета `ru.car.service.message.telegram` и его подпакетов.**

Уровень покрытия проверяется вручную при review. CI-гейт по покрытию в 2.1 не вводим (это отдельная задача T*).

- ☐ Spring-context стартует без `@PostConstruct`-bootstrap в `TelegramConfig` (остаётся только `bot.init()`).

- ☐ `TelegramBotService` не импортирует `NotificationRepository` (и любой другой `*Repository`).

## 9. Риски и план отката

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Регрессия на проде при перекладывании switch в сцены | Средняя | Preserve-behaviour подход; manual smoke на dev перед релизом; ветка feature, merge только после ревью. |
| `MessageSource` не поднимается / ключи не находятся | Низкая | `TelegramMessagesTest` ловит отсутствие ключей при сборке. |
| `CallbackData` теряет старые callback'и у уже активных пользователей | Средняя | Старые форматы (`/qr/<uuid>`, `/notification/<uuid>`) — парсер понимает оба формата (legacy + new) в переходный период; в 2.2 legacy отрезается. **Добавляется явная legacy-обработка в CallbackData.parse.** |
| `MonitoringService.get*CountToday()` окажется нужен где-то ещё | Низкая | При реализации — grep по всему backend'у; удалять только если нет других вызовов. |

**План отката:** всё в одной feature-ветке; merge в main одним коммитом (или squash). Если на проде всплывает регрессия — `git revert` коммита + redeploy. Поскольку БД-миграций в 2.1 нет, откат чистый.
