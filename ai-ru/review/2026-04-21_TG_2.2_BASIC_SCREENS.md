# Спек 2.2 — Telegram-бот: перенос базовых экранов

**Статус:** 📋 Design · **Заведён:** 2026-04-21 · **Эпик:** [2026-04-21_TELEGRAM_EPIC.md](2026-04-21_TELEGRAM_EPIC.md)
**Предусловие:** спек 2.1 замёржен в main (commit `16402a9`).

## 1. Цель

Превратить бот из preserve-behaviour скелета (2.1) в полноценный альтернативный клиент с шестью экранами: Home / Мои метки / Детали метки / Уведомления / Настройки каналов / Профиль. HTML-форматирование, эмодзи, inline edit-in-place навигация.

**Видимое пользователю:** бот радикально меняет UX. Старая reply-клавиатура исчезает, появляется меню с inline-кнопками, экраны переключаются редактированием одного сообщения.

## 2. Архитектура 2.2

Вся инфраструктура 2.1 переиспользуется: `TelegramTransport`, `TelegramRouter`, `SceneRegistry`, `TelegramRenderer`, `CallbackData`, `SceneOutput`, `TelegramMessages`.

**Новое:**

- **6 новых сцен** в `scene/impl/`:
  - `HomeScene` (переписан из заглушки `HomeMenuScene`).
  - `QrListScene` (переписан — карточки вместо плоского списка).
  - `QrDetailsScene`.
  - `NotificationListScene`.
  - `NotificationSettingsScene`.
  - `ProfileScene`.
- **Остаются как есть в 2.1:**
  - `TemporaryQrScene` — меняет только способ отправки: теперь `sendPhoto` с PNG + caption (вместо текст-ссылки).
  - `NotificationMarkReadScene` — используется при клике на элемент списка уведомлений (рендер остаётся прежний до 2.5).
- **HTML-режим:** все новые сцены ставят `SceneOutput.parseMode = "HTML"`.
- **Backquiver:** `TelegramScene` получает дефолт-метод `parentKey()` → `"home"`. Сцены могут override'ить. Callback `<scene>:back` роутер преобразует в вызов `parentSceneKey` через `render(ctx)`.
- **`ReplyKeyboardRemove`** после welcome-сообщения — убираем reply-клаву, показываем inline Home.
- **Транспорт:** `TelegramTransport.sendPhoto(SendPhoto)` — новый метод, реализуется в `TelegramBotService`. Для PNG-отправки QR-кодов.
- **Renderer:** расширяем `SceneOutput` опциональным полем `photo: InputFile` + `caption: String`. Если `photo != null`, рендерер строит `SendPhoto` и шлёт отдельным сообщением через `TelegramTransport.sendPhoto`. Это сохраняет единый API сцен (всё возвращается из `render`/`handle` в виде `SceneOutput`).
- **Утилита:** `MessageUtils.escapeHtml(String)` — escape `<`, `>`, `&` в пользовательском контенте (имена меток, reason-описания).

**Используем существующее:**
- `QrUtils.generateQRCodeImage(url)` → `Resource` с PNG (ZXing + car-id логотип в центре).
- `QrUtils.getQrUrlById(home, id)` → строка `{home}/qr/{id}`.

## 5. Legacy-миграция

- **Reply-keyboard убирается** так: `TelegramAuthorizationService.handle(...)` возвращает `SceneOutput.send(welcomeText, new ReplyKeyboardRemove(true))` — это уже существующий `send(text, replyKeyboard)`-фабричный метод. Роутер после него дополнительно рендерит `HomeScene.render(ctx)` отдельным сообщением (в плане — отдельный код в router.handleText/handleAuth после auth). Итог: юзер видит два сообщения — welcome без клавы + inline Home.
- **Legacy text-triggers:** `canHandleText` для текстов `"QR-коды"` и `"Временный QR"` в `QrListScene` / `TemporaryQrScene` остаётся. Если юзер наберёт эти строки вручную — сработает. Compat-сеть на случай, что у кого-то в чате сохранилась старая reply-клава.
- **Неизвестный текст** после авторизации: **новый `HomeScene` содержит метод `renderUnknown(ctx)`** (как раньше был в `HomeMenuScene`). Вызывается роутером, возвращает сообщение «Неизвестная команда» + inline-кнопка «🏠 Главное меню» (переход на полноценный Home). Старый класс `HomeMenuScene` **удаляется**; его метод `mainKeyboard()` (reply-keyboard) больше никому не нужен — весь UX на inline.

## 10. Риски

| Риск | Митигация |
|------|-----------|
| Сломан edit-in-place при отправке юзером произвольного текста (app-сообщение теряет «свежесть») | `renderUnknown` отвечает новым сообщением с кнопкой «🏠 Главное», которое становится новым «app»-сообщением. |
| Callback 64-байт budget переполнится при фильтрах | Фильтр кодируется через компактное `:qr:<uuid>` в хвосте. UUID = 36 байт + prefix `notif_list:open:all:1:qr:` = 25 → 61 байт. Влезает впритык. План B: ввести отдельный `:f:<uuid>` если нужно урезать. |
| PNG-генерация через `QrUtils.generateQRCodeImage` использует `car-id.png` из classpath — если отсутствует, ImageIO вернёт null | Тест при реализации проверит, что файл есть в `resources/png/`. |
| `UserService.deactivate` нетривиален — логика «что делать с метками пользователя» | Смотрим существующий mobile-flow `AppController.deleteAccount`. Если там просто `user.active=false` — копируем. Если каскад — реализуем тот же каскад в бэке (метод уже может существовать; проверю в реализации). |
| `NotificationSettingRepository.updateTelegramDialogIdByUserId` может не принимать `null` | Если метод объявлен `Long → long` — добавить overload или новый метод `clearTelegramDialogId(Long userId)`. Решим в реализации. |
