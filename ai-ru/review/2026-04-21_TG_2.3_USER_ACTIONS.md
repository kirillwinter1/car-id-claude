# Спек 2.3 — Telegram-бот: пользовательские действия (Report / Support / Marketplace)

**Статус:** 📋 Design · **Заведён:** 2026-04-23 · **Эпик:** [2026-04-21_TELEGRAM_EPIC.md](2026-04-21_TELEGRAM_EPIC.md)
**Предусловие:** спек 2.2 замёржен в main (commit `a7bea10`).

## 1. Цель

Добавить три интерактивных сцены, повторяющих соответствующие экраны мобильного приложения:

- **`ReportEventScene`** — пользователь может сообщить о событии по своему QR (выбор метки → причина → комментарий → отправка).
- **`SupportScene`** — пользователь пишет в поддержку (одно поле — сообщение).
- **`MarketplaceScene`** — ссылки на Ozon/WB для заказа стикеров.

После 2.3 пользователь не зависит от мобильного приложения для всех основных действий: посмотреть/отключить метки, получить/прочитать уведомления, сообщить о событии, обратиться в поддержку, заказать стикеры.

**Видимое пользователю:** три новые кнопки появляются в `HomeScene`. «Сообщить о событии» также добавляется в `QrDetailsScene` (быстрый путь с известным `qrId`).

## 2. Архитектура 2.3

Вся инфраструктура 2.1–2.2 переиспользуется: `TelegramTransport`, `TelegramRouter`, `SceneRegistry`, `TelegramRenderer`, `CallbackData`, `SceneOutput`, `TelegramMessages`, `TelegramScene.parentKey()`.

**Новое — минимум:**

- **3 новых сцены** в `scene/impl/`: `ReportEventScene`, `SupportScene`, `MarketplaceScene`.
- **`SceneStateRegistry`** (`scene/state/`) — маленький in-memory сервис для ожидания текстового ответа пользователя в рамках multi-step форм. Paste'n-drop; stateless callback_data принцип не нарушается (state живёт только для текущего pending ввода, рестарт = retry).
- **`TelegramRouter.handleText`** — перед `SceneRegistry.findByText` проверяет `SceneStateRegistry.poll(chatId)`: если есть pending-ввод → делегирует через `scene.handleText(text, ctx, args)`.
- **`TelegramScene.handleText`** — новый default-метод `SceneOutput handleText(String text, TelegramUpdateContext ctx, List<String> args) { return SceneOutput.noop(); }`.
- **`HomeScene`** — добавить 3 кнопки: «📝 Сообщить о событии», «💬 Поддержка», «🛒 Где купить стикеры».
- **`QrDetailsScene`** — добавить кнопку «📝 Сообщить о событии» (переводит в `ReportEventScene` с предзаполненным `qrId`).

**Используем существующее:**
- `NotificationFacade.send(NotificationDto)` — отправка события.
- `ReasonDictionaryService.findAll()` — список причин.
- `FeedbackFacade.send(FeedbackDto)` — создание feedback + пересылка в admin-канал.
- `FeedbackChannels.TELEGRAM` — enum-значение уже есть.
- `MarketplaceService.get()` — singleton-row с WB+Ozon URLs.
- `QrService.findByUserId(userId)` — список меток пользователя.

## 3. Карта навигации

```
HomeScene
  ├── 📝 Сообщить о событии    ──► ReportEventScene (start)
  │                                 ├── выбрать QR из списка
  │                                 ├── выбрать причину
  │                                 ├── ввести текст (pending-state → handleText)
  │                                 └── отправить → NotificationFacade.send
  │
  ├── 💬 Поддержка               ──► SupportScene (start)
  │                                 ├── ввести текст (pending-state → handleText)
  │                                 └── отправить → FeedbackFacade.send
  │
  └── 🛒 Где купить стикеры     ──► MarketplaceScene (open)
                                      └── inline-кнопки с url= на WB/Ozon

QrDetailsScene
  └── 📝 Сообщить о событии    ──► ReportEventScene (start с args=[qrId])
                                    ├── причина (QR уже выбран)
                                    ├── текст
                                    └── отправка
```

## 4. Экраны

### 4.1 ReportEventScene `report`

**Шаг 1 — выбор QR (если qrId не передан):**

```
📝 <b>Сообщить о событии</b>

Выберите метку, о которой сообщаете:

[ 🟢 Audi Q5 · №145 ]
[ 🟢 Skoda Octavia · №146 ]
[ ⏸ Daewoo · №82 ]
[ ← Назад ] [ 🏠 Главное ]
```

**Шаг 2 — выбор причины (после выбранного QR):**

```
📝 <b>Сообщить о событии</b>
Метка: <b>Audi Q5</b> · №145

Выберите причину:

[ Работает сигнализация ]
[ Мешает проезду ]
[ ДТП ]
[ Ремонт завершён ]
[ Другое ]
[ ← Выбрать метку ] [ 🏠 Главное ]
```

**Шаг 3 — ввод текста:**

```
📝 <b>Сообщить о событии</b>
Метка: <b>Audi Q5</b> · №145
Причина: <b>Работает сигнализация</b>

Напишите текстом, что произошло:
```

(следующим сообщением от пользователя — сам текст. `SceneStateRegistry.put(chatId, "report", "text", [qrId, reasonId])` на этом шаге.)

**Шаг 4 — подтверждение → отправка:**

```
📝 <b>Сообщить о событии</b>

Метка: <b>Audi Q5</b> · №145
Причина: <b>Работает сигнализация</b>
Комментарий: «Сработала, пошла 5 минут назад»

[ ✅ Отправить ] [ ✏️ Изменить комментарий ]
[ ← Изменить причину ] [ 🏠 Главное ]
```

**Callback-контракт:**

```
report:start                              — из Home, без предзаполненного QR
report:start:<qrId>                       — из QrDetails, QR предзаполнен — сразу шаг 2
report:qr:<qrId>                          — выбрали QR → шаг 2
report:reason:<qrId>:<reasonId>           — выбрали причину → шаг 3 (ставит pending-state)
report:preview:<qrId>:<reasonId>:<text64> — НЕ используется, текст хранится в state; из pending-state → шаг 4
report:edit_text:<qrId>:<reasonId>        — шаг 4 → вернуться к вводу текста (pending-state)
report:submit                             — шаг 4 → отправка
report:back                               — вернуться к parent (Home)
```

**Ограничение 64-байт callback_data:** UUID=36 символов + `report:reason:` = 14 + `:` + long-reasonId (≤ 5) = ~56 байт — в бюджете.

**Хранение текста пользователя:** в `SceneStateRegistry` (не в callback_data — плохая идея для многострочного текста). После `report:reason:...:...` state = `{scene:"report", action:"text", args:[qrId, reasonId]}`. Пользователь пишет текст → `handleText(text, ctx, [qrId, reasonId])` → формируем preview-экран; **в этом же state dump'е остаётся и сам введённый text** в поле `draftText`. Кнопка «✅ Отправить» делает `report:submit` — сцена достаёт state (не очищая), берёт `draftText` + args, вызывает `NotificationFacade.send(new NotificationDto(qrId, reasonId, draftText, senderId=ctx.userId()))` → clear state.

**Успех:** заменить сообщение на «✅ Событие отправлено» + кнопка «🏠 Главное».
**Ошибка:** «❌ Не удалось отправить: {причина}» + кнопка «← Вернуться».

### 4.2 SupportScene `support`

**Шаг 1 — приглашение:**

```
💬 <b>Поддержка</b>

Опишите проблему или предложение — следующим сообщением. Мы ответим на телефон/Telegram, привязанный к аккаунту.
```

(ставит `SceneStateRegistry.put(chatId, "support", "text", [])`.)

**Шаг 2 — после текста:**

```
💬 <b>Спасибо!</b>

Ваше обращение принято. Мы напишем в ближайшее время.

[ 🏠 Главное ]
```

**Callback-контракт:**

```

support:back    — parent (Home)
```

**Текст-ввод:** `handleText(text, ctx, [])` → формирует `FeedbackDto{email=ctx.user().getPhoneNumber()+"@telegram", text=text, channel=TELEGRAM, userId=ctx.userId()}` → `FeedbackFacade.send(dto)` → clear state → шаг 2.

*Email формально обязателен в DTO (`@NotBlank`). Синтезируем «{phone}@telegram», чтобы не вводить ещё одно поле. Смысловая нагрузка — всегда понятно, откуда feedback (в канал летит `TelegramBotService.sendFeedback(from, text)`, где `from = dto.getEmail()`; admin-канал увидит «79001234567@telegram»). Если понадобится — позже можно добавить правило-маппинг в `TelegramBotService.sendFeedback`, но в 2.3 это overkill.*

## 5. SceneStateRegistry

**Назначение:** удерживать короткоживущий state между шагом «попросили ввести текст» и «пользователь ответил».

**API:**

```java
package ru.car.service.message.telegram.scene.state;

@Service
public class SceneStateRegistry {

    public record PendingText(String scene, String action, List<String> args, String draftText, long expiresAt) {}

    private static final Duration TTL = Duration.ofMinutes(5);
    private final Map<Long, PendingText> states = new ConcurrentHashMap<>();

    public void put(long chatId, String scene, String action, List<String> args) { ... }
    public void updateDraft(long chatId, String draftText) { ... }       // сохранить текущий draft
    public Optional<PendingText> peek(long chatId) { ... }                // читать без удаления (+evict expired)
    public Optional<PendingText> pop(long chatId) { ... }                 // читать и удалить
    public void clear(long chatId) { ... }
}
```

**Принципы:**
- In-memory, не в БД. Рестарт = потеря pending; user retry — ok.
- TTL 5 минут. Автоочистка при чтении (`peek`/`pop` evict'ят expired).
- Concurrency-safe (`ConcurrentHashMap`).
- Нет шардинга: бот single-instance (см. «Что НЕ делаем» в эпике).

## 6. Роутер — `handleText` изменяется

```java
private void handleText(TelegramUpdateContext ctx) {
    String text = ctx.text().orElse("");
    Optional<SceneStateRegistry.PendingText> pending = sceneStateRegistry.peek(ctx.chatId());
    if (pending.isPresent()) {
        Optional<TelegramScene> scene = sceneRegistry.findByKey(pending.get().scene());
        if (scene.isPresent()) {
            SceneOutput output = scene.get().handleText(text, ctx, pending.get().args());
            renderer.dispatch(output, ctx.chatId(), null);
            return;
        }
    }
    // старая логика: findByText → fallback
    ...
}
```

**Нюанс**: сцена сама решает, очистить state или обновить (например, `ReportEventScene.handleText` сохраняет draftText и держит state до submit; `SupportScene.handleText` сразу submit'ит и чистит state).

## 8. Ограничения и уточнения

- **Маркетплейсы не кеш-инвалидируются**: `MarketplaceService.get()` читает БД на каждый клик. Одна запись, запрос быстрый — ok.
- **ReasonDictionary не имеет иконок** (см. [TECH_DEBT A13](../TECH_DEBT.md)): в 2.3 кнопки причин — чистый текст. Эмодзи по типу причины — в 2.5 с рендером карточки уведомления.
- **Список меток для отчёта** — используем `QrService.findByUserId(userId)`. Фильтруем по `QrStatus.ACTIVE` и `TEMPORARY` (о `DELETED` / `NEW` нет смысла сообщать). Если список пуст → `tg.report.empty_qrs` и кнопка «🏠 Главное».
- **Validation NotificationDto**: `qrId` и `reasonId` обязательны; `text` — до 500 символов (см. `NotificationDto` @Size). При превышении — обрезаем в 500 в `ReportEventScene.handleText` + предупреждаем пользователя.
- **`sendFeedback` → admin-канал** — формат `from:\ntext`. В `TelegramBotService.sendFeedback` как есть. Можем добавить hint «[via TG bot]» позже, в 2.3 не трогаем.
- **Авторизация**: `NotificationFacade.send` и `FeedbackFacade.send` ожидают populated `senderId`/`userId`. Мы берём из `ctx.userId()` — оба доступны в Telegram, т.к. после 2.1 у нас всегда аутентифицированный контекст.
- **Concurrency**: один пользователь может параллельно начать `report` и `support`? SceneStateRegistry хранит по `chatId`, значит — только одну pending-операцию. Вторая перезапишет. UI-поток линейный, конфликтов не должно быть. Документировать — не надо, стандартное поведение.

## 9. Что НЕ делаем в 2.3

- **Подтверждение телефона при поддержке.** Берём из user автоматически.
- **Прикрепление фото/файлов в `report`/`support`.** TG позволяет, но мобилка такого не имеет, значит и бот пока нет (возможно в 2.5).
- **Deep-link onboarding** — это 2.4.
- **Уведомления-карточки** — это 2.5.
- **Эмодзи по causes** — тоже 2.5 (с рендером уведомлений).
- **Smart-retry / dead-letter для failed `NotificationFacade.send`** — stub (просто показываем ошибку пользователю). Кандидат в [TECH_DEBT A18](../TECH_DEBT.md).
- **Multi-instance SceneStateRegistry** — single process, single node.
