# BF5: «Позвонить владельцу» — раскрытие номера прохожему

**Статус:** 📋 Planned (после стабилизации; Тир 2 уже выкачен 2026-06-27 → разблокировано)
**Последний апдейт:** 2026-06-27 (дизайн уточнён в брейншторме)
**Инициатор:** отзыв покупателя на Ozon («у ваших конкурентов сразу выпадает выбор связи с владельцем… позволить владельцу метки решить, готов ли он показать свой номер») + продуктовый шторм 2026-06-14.

## Что делаем

Даём прохожему возможность **позвонить владельцу напрямую** по `tel:`-ссылке на экране статуса веб-отправки. **Модель C** (владелец раскрывает свой реальный номер по согласию). Модели A (анонимный релей-чат) и B (прохожий оставляет свой номер) — не делаем (отдельные будущие карточки).

## Решения брейншторма (2026-06-27)

1. **Реальный номер** владельца (`tel:+7…`), не маскированный. Маскированный (виртуальный, Zvonok) — план Б при злоупотреблениях.
2. **Когда показывать:** через **~60с после отправки**, если у владельца настройка ON — **независимо от прочтения** (`READ` номер НЕ скрывает). То есть это не «только при неответе», а «облегчённая связь после короткой задержки».
3. **Дефолт настройки:** **ON для новых** пользователей (с явным согласием при регистрации), **OFF для существующих** (~1830 — согласия не давали; включают сами тумблером).

## Поток (одна веб-отправка)

1. **t=0** — прохожий отправил уведомление → push/Telegram владельцу (текущий флоу [F4](../features/F4_WEB_REPORT.md)/[F5](../features/F5_NOTIFICATIONS.md)).
2. **t≈30с** — если `UNREAD` и у владельца включены звонки → робот-звонок (Zvonok [F11](../features/F11_ZVONOK.md), уже есть).
3. **t≈60с** — если у владельца `show_phone_on_unreachable = true` → на экране статуса появляется **«Позвонить владельцу: +7…»** (`tel:`). Показывается **и при `UNREAD`, и при `READ`**.

## Как отдаём номер (переиспользуем Тир 2)

Не новый эндпоинт, а расширяем существующий **`GET /api/notification/{id}/status`**: добавляем nullable-поле `owner_phone`. Сервер заполняет его **только** при выполнении условий: уведомление существует, прошло ≥ порога (~60с) с `created_date`, у владельца настройка ON. Фронт уже опрашивает status (Тир 2) — переиспользуем, новый API не плодим.

**Антискрейпинг:** номер доступен только по `notificationId` (создаётся реальной веб-отправкой, есть rate-limit на отправку), **не** из сырого `GET /api/qr/{uuid}`. Доп. rate-limit на status при необходимости. ⚠️ Поскольку убран гейтинг «только при неответе», для ON-владельцев номер показывается практически на каждой отправке после задержки — поэтому критичны дефолт-OFF для старых и согласие для новых.

## Данные

- `notification_settings.show_phone_on_unreachable BOOLEAN` — миграция Liquibase, **default FALSE** (существующие строки = OFF).
- Новым пользователям ставим **TRUE** при создании настроек (в коде `UserService`/создание `notification_settings`, не через DB default).
- Поле — в модель `NotificationSetting`, INSERT/UPDATE, `NotificationSettingDto`.

## Управление владельцем

- Тумблер **«Показывать мой номер прохожему, если мы не смогли с вами связаться»** (честная положительная формулировка) — в настройках **мобайла** и **Telegram-бота**, через существующий update настроек. Лёгкий opt-out.
- Per-метка granularity, Telegram-ник вместо номера — на будущее.

## Согласие (новые пользователи)

При регистрации — явное согласие/уведомление (152-ФЗ), что номер может быть показан прохожему. Текст — за владельцем проекта (юр.). Без этого дефолт-ON для новых не включать.

## Фазы реализации

- **Фаза 1 (backend + web):** колонка + миграция; `owner_phone` в `/notification/{id}/status` с гейтингом (порог + настройка); блок «Позвонить владельцу» на `notification.html`.
- **Фаза 2 (управление + согласие):** тумблер в мобайле и Telegram-боте; согласие при регистрации; дефолт TRUE для новых.
- ⚠️ Зависимость: для существующих (дефолт OFF) фича «оживает» только после Фазы 2 (тумблер); для новых — после согласия. Ценность даёт Фаза 1+2 вместе.

## Тесты

- Гейтинг: `owner_phone` отдаётся при (прошёл порог + настройка ON); НЕ отдаётся при (рано / настройка OFF). Проверка, что `READ` не влияет.
- Дефолты: существующий пользователь = OFF, новый = ON.
- Антискрейп: номер не приходит из `GET /api/qr/{uuid}`; rate-limit.
- Миграция.

## ⚠️ Остаточный риск (зафиксирован осознанно)

Номер показывается **даже при прочтении** → раскрывается **чаще**, чем в исходной «только при неответе» версии → выше экспозиция личного номера.
- **Юридически:** зона 152-ФЗ. Митигации: **дефолт OFF для старых**, **явное согласие для новых**, честный лейбл, лёгкий opt-out, доступ только по `notificationId` + rate-limit.
- **Реальный номер, раз показанный, прохожий может сохранить** — плата за выбор «реальный номер».
- **План Б при спаме/жалобах:** маскированный (виртуальный) номер через Zvonok — связь без раскрытия реального номера.

## Открытые вопросы
