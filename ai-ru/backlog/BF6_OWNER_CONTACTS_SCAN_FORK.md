# BF6: Контакты владельца + развилка действий на скане

**Статус:** 📋 Planned (расширяет/включает контактную часть [BF5](BF5_OWNER_PHONE_REVEAL.md))
**Последний апдейт:** 2026-07-01 (брейншторм)
**Инициатор:** продолжение BF5 + запрос владельца: дать прохожему сразу выбор — сообщить о событии, написать в мессенджер или позвонить. Ozon-отзыв: «у конкурентов сразу выпадает выбор связи с владельцем».

## Что делаем

После скана метки прохожий видит **развилку действий** (а не сразу форму события). Владелец публикует контакты (телефон + Telegram/VK/MAX), и прохожий может связаться напрямую — или сообщить о событии как раньше.

## Экран метки после скана (`qr.html`) — развилка

Кнопки показываются по тому, что владелец опубликовал:
1. **«Сообщить о событии»** (основная) → создаёт черновик → `notification.html` (выбор причины) — текущий флоу. *(Сейчас черновик создаётся при загрузке `qr.html`; переносим на клик по этой кнопке.)*
2. **«Написать в Telegram»** → `t.me/<username>` (если задан).
3. **«Написать в VK»** → `vk.me/<username|id>` (если задан).
4. **«Написать в MAX»** → `max.ru/<...>` (если задан).
5. **«Позвонить»** → `tel:+7…` (если владелец включил показ номера).

Проверка статуса метки (ACTIVE/TEMPORARY vs NEW/DELETED) — как сейчас.

## Решения брейншторма (2026-07-01)

1. **Контакты видны сразу на скане** (не гейтим за отправкой уведомления). Защита приватности — **неугадываемый UUID метки (v4, перебором не собрать)** + **opt-in владельца** (пусто = кнопки нет).
2. **Телефон** — кнопка «Позвонить» сразу (как мессенджеры), по флагу владельца. **Дефолт: ON для новых** пользователей (нужно согласие/Политика), **OFF для существующих**.
3. **Мессенджеры** — opt-in по наличию значения (пусто = не показываем). Дефолт пусто.
4. **Каналы:** Telegram (`@username`), VK (ссылка/id), MAX (**вставить ссылку** из профиля — у MAX нет @username, формат `max.ru/u/<хеш>`).

## Backend

- **Модель `notification_settings`** += `telegram_contact`, `vk_contact`, `max_contact` (varchar, nullable). Телефон — существующий `show_phone_on_unreachable` (переосмысливаем как «публиковать номер»; берём `User.phoneNumber`). Миграция **1.19**.
- **`GET /api/qr/{uuid}` (`QrDto`)** += объект `owner_contacts: { phone, telegram, vk, max }` — только опубликованное, иначе `null`. Ссылки бэкенд собирает готовыми (`t.me/…`, `vk.me/…`, `max.ru/…`, `+7…`). Наполняет `QrService.getQrById` из настроек/юзера владельца метки.
- **Нормализация ввода** (бэкенд строит URL): Telegram — из `@username`/`t.me/...`; VK — из `vk.com/...`/id; MAX — принимаем ссылку `max.ru/...` как есть. Хранить нормализованное значение.
- **Новые пользователи:** `show_phone_on_unreachable=true` при создании настроек (см. BF5 Фаза 2).

## Frontend (web)

- **`qr.html` + `script.js`:** вместо авто-редиректа рендерим развилку из `owner_contacts`. «Сообщить о событии» → текущий createDraft-флоу. Мессенджеры/звонок → ссылки.
- Стиль кнопок — по образцу `.owner-call-button` (уже есть).

## Mobile (один релиз)

- Экран настроек: тумблер «Показывать мой номер» + 3 поля ввода (Telegram `@username`, VK ссылка/id, MAX — вставить ссылку). Уходит через `notification_settings.patch`.
- Модель `NotificationSettings` += `telegramContact`/`vkContact`/`maxContact`; контроллер + экран.
- Версия `1.1.4+44 → 1.1.5+45`. **Сборку и публикацию в RuStore/App Store делает владелец** (нет тулчейна/ключей/аккаунтов у ассистента).

## Согласие / 152-ФЗ

- Контакты — opt-in (владелец сам публикует). Телефон дефолт-ON для новых → **обновить Политику/Оферту** (`car-id.ru/privacy-policy`, `/offer`) до включения дефолт-ON на проде. Чекбокс согласия при регистрации уже есть.
- Мессенджер-хэндлы менее чувствительны (публичны на своих платформах).

## Связь с BF5

- BF5 Фаза 1 (показ номера на странице **статуса** после отправки) уже в `main` (выключено). BF6 — контакты на **скане** (новый основной путь). Не конфликтуют; статус-страница остаётся пост-отправочным касанием.
- BF5 Фаза 2 (мобильный тумблер телефона) **поглощается** BF6 (тумблер телефона + поля мессенджеров в одном экране/релизе).

## Фазы реализации

- **Фаза A (backend):** миграция 1.19 (contact-поля) + `owner_contacts` в `QrDto`/`QrService` + нормализация + патч настроек принимает новые поля + дефолт-ON телефона для новых.
- **Фаза B (web):** развилка на `qr.html`.
- **Фаза C (mobile):** поля контактов + тумблер, bump версии → релиз (владелец).
- ⚠️ Дефолт-ON телефона деплоить после обновления Политики.

## Тесты

- Backend: `owner_contacts` отдаётся с опубликованными каналами / `null` для пустых; нормализация ссылок; патч сохраняет contact-поля; дефолт-ON телефона для новых; миграция.
- Web: развилка рендерит кнопки по наличию контактов; «Сообщить» ведёт в текущий флоу.
- Mobile: fromJson новых полей; ручная проверка UI.

## Риски / открытые вопросы

- **Приватность:** контакты публичны для любого, кто открыл ссылку метки (не только физически отсканировал — если ссылку переслали/проиндексировали). Митигация: opt-in, неугадываемый UUID; при жалобах — маскированный номер (Zvonok) как план Б.
- MAX: ссылка открывает чат только если профиль публичный/в контактах — прохожий-незнакомец может упереться в приватность MAX. Отметить в UX-подсказке владельцу.
- Валидация ввода контактов (чтобы не публиковать мусор/чужие ссылки).
- Точные тексты кнопок и подсказок владельцу.
