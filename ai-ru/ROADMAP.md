# Roadmap Car-ID

> Путь развития проекта от сегодняшнего состояния. Точки **не** по датам, а по порядку: переходим к следующему этапу только когда предыдущий закрыт.

## Контекст

Бизнес-основа: продажа QR-стикеров на Ozon (`~1.5 года продаж, объёмы небольшие`) + мобильное приложение для владельцев авто. Аналитикой и операционкой маркетплейса занимается соседний проект [`ozon-workspace`](../../ozon-workspace/) (MCP-сервер Ozon Seller API). Любые продуктовые решения здесь должны учитывать сигналы оттуда — отзывы, конверсию, аудиторию.

Связанная документация:
- [FEATURES.md](FEATURES.md) — что уже реализовано.
- [TECH_DEBT.md](TECH_DEBT.md) — технический долг.
- [backlog/](backlog/) — карточки запланированных фич (BF).
- [review/](review/) — архитектурные ревью и рефакторинги.

## Phase 3 — Мульти-канальная авторизация и уведомления 📋

**Фокус:** снять зависимость от платного flashcall, упростить вход, добавить каналы получения уведомлений.

Подробности → [backlog/BF2_MULTI_CHANNEL_AUTH.md](backlog/BF2_MULTI_CHANNEL_AUTH.md).

Вкратце:
- Авторизация через Telegram (Login Widget / oauth bot).
- Авторизация через MAX messenger.
- Возможно: VK ID / «войти через Яндекс».
- Единое API для выбора канала получения уведомлений (push / Telegram / MAX / flashcall — опционально).
- В этой же фазе закрываем [TECH_DEBT P1](TECH_DEBT.md) — ротация `token.signing.key`, вынос секретов с диска, dual-key механизм для безопасной ротации без массового релогина.

**Предусловие:** Phase 2 завершена (Telegram-бот готов).

## Phase 5 — Аренда машиномест в ЖК 🏗️ (эпик)

**Фокус:** превратить Car-ID из уведомительного сервиса в маркетплейс парковочных мест в жилых комплексах. Референс — [Казимир](https://kazimir-carplace.ru) (Telegram-бот @Car_Place_bot).

Подробности → [backlog/BF4_PARKING_RENTAL_EPIC.md](backlog/BF4_PARKING_RENTAL_EPIC.md).

**Почему именно нам имеет смысл:**
- Целевая аудитория Car-ID (владельцы авто в ЖК) = аудитория фичи.
- QR-стикер уже живёт на машине, можно использовать как «якорь» присутствия в ЖК (сосед увидел стикер → узнал об аренде мест через тот же сервис).
- Telegram-бот уже готов (Phase 2) — основа UX для аренды.
- Мультиканальная авторизация (Phase 3) — нужна для быстрого онбординга новых арендодателей.
- Есть готовый фундамент платёжной интеграции через `ozon-workspace`.

**Высокоуровневая разбивка на под-фичи** (детали в BF4):
- «ЖК» как сущность (справочник, привязка к локации).
- «Машиноместо» с состоянием (свободно / забронировано / занято).
- Роли: арендодатель, арендатор.
- Поиск мест по дате/времени и ЖК.
- Эскроу-оплата (карта → СБП, деньги владельцу после окончания аренды).
- Рейтинги и отзывы между сторонами.
- Разрешение споров (опционально).

**Предусловие:** Phase 4 завершена + продуктовая валидация (спрос, ЖК-пилоты).

## Принципы движения по roadmap

1. **Никогда не перескакиваем фазы.** Фаза N+1 стартует только когда N закрыта; правило зафиксировано в памяти Claude (`project_current_phase.md`).
2. **Каждая фаза открывается `review/`-документом** с полным планом, рисками и критериями завершения. Сейчас есть только `review/2026-04-21_DOCKERIZATION.md` — он прицеплен к Phase 3 (попадёт в общий большой деплой).
3. **Бизнес-ценность первична** — см. memory `feedback_business_value_first.md`. Перед каждой подзадачей проверяем: что это даёт пользователю / бизнесу / сколько стоит.
4. **Prod — read-only**, изменения только с явным подтверждением (memory `feedback_prod_readonly.md`).
